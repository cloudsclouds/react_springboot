# 工作台语音能力技术方案（MVP）

> 本文档定义工作台页面（`src/pages/WorkspaceHomePage.tsx`）中新增的语音能力 MVP 方案。当前目标是先在工作台页面完成一个轻量、可用、易扩展的录音播放器 + 语音输入模块。

## 1. 目标

在工作台页面中新增一组语音相关能力，用于支持录音回放、语音输入和基础转写展示，满足以下场景：

1. 录音的音频回放与检查
2. 语音对话或语音内容采集
3. 转写文本的快速查看与复制
4. 断点续播，提升重复听音效率

## 2. 本期范围

- 录音播放器
  - 音频播放 / 暂停
  - 波形可视化
  - 倍速播放
  - 进度拖拽
  - 断点续播（本地存储）
- 语音输入模块
  - 麦克风录制
  - 录音结束后触发语音转写
  - 转写结果展示在一个 `input` 中
- 页面集成
  - 直接集成到 `src/pages/WorkspaceHomePage.tsx`
  - 详细组件封装到 `src/components/Audio` 文件夹中

## 3. 明确不做的内容

为了保证本期实现轻量可落地，以下能力暂不实现：

- 转写文本导入编辑器
- 播放进度后端持久化
- 录音转写记录表
- 录音质检记录表
- 语音任务列表页
- 复杂的音频审计、质检工作流
- 多端同步恢复播放进度

## 4. 前端技术方案

### 4.1 组件形态

建议在 `WorkspaceHomePage.tsx` 中拆出一个独立的语音区域，内部可按以下结构实现：

- `VoiceRecorderPanel`
  - 开始录音
  - 停止录音
  - 展示录音状态
- `VoicePlayerPanel`
  - 音频播放器
  - 波形展示
  - 倍速控制
  - 断点续播
- `VoiceTranscriptPanel`
  - 转写结果输入框
  - 仅用于查看、复制、编辑，不自动回填到编辑器

### 4.2 录音实现

使用浏览器原生能力完成采集：

- `navigator.mediaDevices.getUserMedia({ audio: true })`
- `MediaRecorder` 负责录音
- 录音完成后生成音频 `Blob`
- 将 `Blob` 转为可播放的 `ObjectURL`

实现原则：

- 录音逻辑尽量独立于页面业务状态
- 录音中应有明显状态提示：未开始 / 录音中 / 已完成 / 转写中 / 转写完成 / 失败
- 若浏览器权限不足，需提供可理解的提示

### 4.3 波形可视化

波形可视化建议采用方案A：

- 方案 A：`wavesurfer.js`
  - 优点：实现快、交互成熟、支持波形与播放联动
  - 适合当前 MVP
- 方案 B：`Web Audio API + Canvas`
  - 优点：可完全定制样式
  - 缺点：实现成本更高

建议本期优先使用 `wavesurfer.js`，以减少开发复杂度并快速验证体验。

### 4.4 倍速播放

播放器至少支持以下倍速档位：

- 0.5x
- 1.0x
- 1.25x
- 1.5x
- 2.0x

实现要求：

- 倍速切换后立即生效
- 断点续播时保留上次倍速设置
- 倍速状态也可放入本地存储，和播放进度一起恢复

### 4.5 断点续播

本期断点续播仅采用 `localStorage`，不做后端持久化。

建议存储内容：

- 音频唯一标识
- 当前播放时间 `currentTime`
- 播放倍速 `playbackRate`
- 最近更新时间 `updatedAt`

恢复规则：

1. 页面重新打开时读取 `localStorage`
2. 如果存在对应音频的播放进度，则在音频加载完成后自动 seek 到该位置
3. 若进度超过音频总时长，则回退到起点
4. 若本地存储异常或音频已失效，则自动忽略并重新从 0 开始

说明：

- 断点续播仅作为本地增强体验，不作为跨设备同步能力
- 如果后续需要跨端恢复，再补后端接口和持久化模型

## 5. 语音输入与转写方案

### 5.1 语音输入流程

推荐流程如下：

1. 用户点击“开始录音”
2. 浏览器申请麦克风权限
3. 录音中实时显示状态
4. 用户点击“停止录音”
5. 前端生成音频数据并提交转写请求
6. 后端返回转写结果
7. 前端将转写结果显示在 `input` 中

### 5.2 转写结果呈现

本期不把转写文本导入编辑器，只使用一个 `input` 展示：

- 便于快速查看和复制
- 便于后续再手工进入编辑器
- 降低当前工作台与编辑器之间的耦合

如果需要增强交互，可在输入框旁增加：

- 一键复制
- 清空
- 手工编辑
- 字数统计

### 5.3 转写请求形态

本期后端只需要“存转写结果”，不要求记录转写任务历史或质检记录。

建议接口行为：

- 前端提交录音文件到后端
- 后端完成转写后直接返回结果文本
- 前端拿到结果后写入 `input`

如果后端采用同步接口不方便，也可以先返回任务结果给前端，前端直接展示，不强制建立任务表。

## 6. 接口设计（本期）

本期接口只保留最小闭环，围绕“录音上传 -> 音频播放 -> 转写展示”展开。

### 6.1 接口 1：录音分片上传（长录音）

- **功能**：前端录音中按 `MediaRecorder.start(timeslice)` 每 1~3 秒上传一个分片
- **用途**：降低长录音内存压力、提高长时录音成功率
- **请求方式**：`POST`
- **建议路径**：`/api/voice/recordings/chunks/upload`
- **请求体**：`multipart/form-data`

字段建议：

- `file`：当前音频分片（`Blob` / `File`）
- `sessionId`：本次录音会话 ID
- `chunkIndex`：分片序号（从 0 递增）
- `isLastChunk`：是否最后一片
- `source`：可选，来源标记，如 `workspace`

返回建议：

```json
{
  "success": true,
  "message": "分片接收成功",
  "data": {
    "accepted": true
  }
}
```

### 6.1.1 接口 1-2：分片上传完成通知（触发合并）

- **功能**：通知后端该 `sessionId` 的分片上传已结束，后端执行合并
- **请求方式**：`POST`
- **建议路径**：`/api/voice/recordings/chunks/complete`
- **请求体**：`application/json`

```json
{
  "sessionId": "1716790000123"
}
```

返回建议：

```json
{
  "success": true,
  "message": "合并成功",
  "data": {
    "audioId": "audio-001",
    "fileName": "recording.webm"
  }
}
```

### 6.2 接口 2：后端返回 Blob 文件，前端可直接播放

- **功能**：前端根据音频标识请求后端，后端返回音频 `Blob`
- **用途**：用于播放器直接播放、波形渲染、断点续播恢复
- **请求方式**：`GET`
- **建议路径**：`/api/voice/recordings/{audioId}/blob`
- **响应类型**：`audio/*` 或 `application/octet-stream`

前端使用方式：

- 通过 `fetch` 拉取二进制数据
- 将响应转成 `Blob`
- 使用 `URL.createObjectURL(blob)` 生成播放地址
- 交给播放器组件播放和渲染波形

### 6.3 接口 3：清空播放进度（防脏数据）

- **功能**：前端点击“清空进度”时，通知后端清理该音频的服务端进度状态（若已实现）
- **请求方式**：`POST`
- **建议路径**：`/api/voice/playback/clear`
- **请求体**：`application/json`

```json
{
  "audioId": "audio-001",
  "source": "workspace"
}
```

返回建议：

```json
{
  "success": true,
  "message": "播放进度已清空",
  "data": true
}
```

### 6.4 后端约束（本期）

后端本期只保留最小能力：

- 接收音频文件
- 保存音频文件
- 根据 `audioId` 返回音频 Blob
- 执行语音转写
- 返回转写结果
- 如有必要，仅存最终转写结果

本期不实现：

- 转写任务记录表
- 录音质检记录表
- 播放进度持久化接口
- 音频审计与检索列表

## 7. 页面集成方案

### 7.1 集成位置

实现位置固定为：

- `src/pages/WorkspaceHomePage.tsx`

### 7.2 页面结构建议

页面可分为三块：

1. 顶部概览区：保留当前工作台概览卡片
2. 中部语音工作区：新增录音播放器与语音输入模块
3. 底部辅助区：可预留状态提示或历史占位

### 7.3 交互原则

- 不改变工作台原有入口含义
- 不增加复杂导航层级
- 语音模块应作为工作台中的一个功能面板存在
- 音频播放和转写输入之间保持低耦合

## 8. 本地存储设计

建议使用以下 key 规则：

- `workspace:voice:progress:{audioId}`：保存播放进度
- `workspace:voice:rate:{audioId}`：保存倍速

可合并为一个对象：

```json
{
  "audioId": "audio-001",
  "currentTime": 128.5,
  "playbackRate": 1.25,
  "updatedAt": 1710000000000
}
```

## 9. 风险与处理

### 9.1 浏览器兼容性

- `MediaRecorder` 在现代浏览器支持较好，但仍需注意 Safari 等环境兼容性
- 若浏览器不支持录音，应提供降级提示

### 9.2 音频格式

- 前端录音格式可能与后端 ASR 偏好格式不一致
- 若后端识别效果不稳定，可在后续增加格式转换层

### 9.3 本地断点续播准确性

- `localStorage` 仅适合单设备单浏览器恢复
- 页面频繁刷新或音频文件变更时，应自动忽略旧进度

### 9.4 录音权限

- 用户拒绝麦克风权限时，页面必须有清晰提示
- 不应让整个工作台因为语音功能失败而不可用

## 10. 实现优先级

推荐优先级如下：

1. 录音与播放基础能力
2. 波形可视化
3. 倍速播放
4. `localStorage` 断点续播
5. 语音转写结果输入框展示
6. 页面样式与细节优化

## 11. 结论

本期方案采用“**前端轻量集成 + 本地断点续播 + 转写结果直显输入框**”的方式，能以最小成本在 `WorkspaceHomePage.tsx` 中快速落地语音能力，并为后续的音频质检、跨端同步和编辑器联动预留扩展空间。
